= Net32 Dental Supply (kitchensink): Spring Boot 3.x Multi-Service Architecture
:toc: left
:toclevels: 3

The `kitchensink` quickstart — the *Net32 Dental Supply* sample — was originally a single
Jakarta EE 10 web application packaged as `kitchensink.war` and deployed on JBoss EAP 8.0.
It has been refactored into *three independently deployable Spring Boot 3.x microservices*
that share one PostgreSQL database. This guide documents the refactored architecture, the
cross-service HTTP contracts, the stored-procedure-to-Java migration, and how to build, run,
test, and ship the system.

This README is organized into nine sections: Overview, Architecture, the
shared-database / separate-ownership model, the cross-service HTTP contracts, the
stored-procedure-to-Java parity, Prerequisites & Build, Running the services, Testing, and
Docker & CI.

== 1. Overview

*Net32 Dental Supply* is a small dental-supply marketplace sample. In its original form it was a
single, layered Jakarta EE 10 WAR (`kitchensink`, `packaging=war`) built against the JBoss EAP
quickstart parent and deployed on JBoss EAP 8.0. Its core business logic lived in six PL/pgSQL
stored procedures, and EJB/CDI service beans invoked those procedures through
`EntityManager.createNativeQuery(...)` rather than expressing the logic in Java.

This project has been re-platformed and decomposed into *three independently deployable Spring
Boot 3.x microservices*, organized by business domain, that together share *one* PostgreSQL
database:

* `marketplace-service` — catalog, vendors, and pricing.
* `orders-service` — carts, orders, shipping, and discounts.
* `users-service` — members and loyalty-tier recalculation.

The migration is a *minimal-change refactor*: every unit of observable behavior and every
externally consumed REST contract is preserved exactly. Only the changes necessary to accomplish
the refactor were made — no features were added or "improved".

Key transformations applied during the migration:

* *Tech stack* — JBoss EAP 8.0 / Jakarta EE 10 WAR became Spring Boot 3.x executable JARs on a
  Java 17 baseline.
* *EJB / CDI / JAX-RS idioms* — `@Stateless` / `@ApplicationScoped` beans became `@Service`
  with constructor injection; JAX-RS `@Path` / `@GET` / `Response` became Spring MVC
  `@RestController` / `@GetMapping` / `ResponseEntity`; the `@Singleton @Startup @Schedule` EJB
  timer became `@EnableScheduling` + `@Scheduled(cron=...)`.
* *Persistence* — hand-rolled `EntityManager` / Criteria / JPQL repositories became Spring Data
  `JpaRepository` interfaces, and the six `EntityManager.createNativeQuery(...)` stored-procedure
  call sites were eliminated (see Section 5).
* *Retained namespace* — the `jakarta.persistence.*` and `jakarta.validation.*` annotations on
  the entity classes are *unchanged*, because Spring Boot 3.x uses the same `jakarta` namespace.
  The ten JPA entities moved package only; their column mappings, field names, validation
  annotations, and relationships are preserved.

== 2. Architecture

The system is a Maven multi-module project rooted at `kitchensink/`. The aggregator parent POM
declares `packaging=pom` and lists three child modules; each child is a self-contained Spring
Boot application. The Java base package retains the historical root
`org.jboss.as.quickstarts.kitchensink` and appends a domain segment — `.marketplace`, `.orders`,
or `.users`.

.The three services and their bounded contexts
[cols="2,1,2,3,2",options="header"]
|===
| Service | Port | Context path | Owned tables | Domain services

| `marketplace-service`
| 8081
| `/marketplace`
| `products`, `vendors`, `vendor_inventory`
| `PricingService`, `VendorSelectionService`

| `orders-service`
| 8082
| `/orders`
| `orders`, `order_items`, `order_draft_items`, `shipping_zones`, `discount_audit`
| `OrderService`, `ShippingService`, `DiscountService`

| `users-service`
| 8083
| `/users`
| `member`
| `MemberRegistration`, `TierRecalculationService`
|===

Within each service the package layout and request flow are identical and layered
*`@RestController` -> `@Service` -> Spring Data `@Repository`*:

* `model/` — JPA entities (the `jakarta.persistence` / `jakarta.validation` annotations are retained).
* `repository/` — Spring Data `JpaRepository<T, ID>` interfaces (derived queries plus `@Query`).
* `service/` — `@Service` beans holding the extracted business logic.
* `rest/` — `@RestController` classes exposing the HTTP API.
* `client/` — thin `@Component` HTTP gateways to other services (orders and users only).
* `dto/` — producer/consumer DTOs for the cross-service contracts.
* `exception/` — domain exceptions mapped to HTTP status codes.

The frontend is a server-rendered PHP storefront under `frontend/` that calls the three services
over HTTP. The runtime topology and the cross-domain call graph are:

....
                 PHP frontend (frontend/*.php)
                          |  HTTP
        +-----------------+------------------+
        v                 v                  v
  marketplace-service  orders-service    users-service
     :8081                :8082             :8083
   /marketplace          /orders           /users
        ^                 |  |                 |
        |  price (C1)     |  | tier (C2)       | spend (C3)
        +-----------------+  +----> users      +----> orders
                          |
        all three  <----- shared PostgreSQL "kitchensink" (ddl-auto=validate)
....

The defining rule is deliberate: although all three services share one physical PostgreSQL
database, any data a service does *not* own is obtained *over HTTP*, never through a SQL join or
a shared compile-time dependency. This makes the domain boundary explicit and independently
evolvable.

== 3. Shared-database / separate-ownership model

All three services connect to *one* PostgreSQL database named `kitchensink`:

[source,properties]
----
spring.datasource.url=jdbc:postgresql://localhost:5432/kitchensink
spring.jpa.hibernate.ddl-auto=validate
----

The decomposition follows a *shared-database, separate-ownership* model (not
database-per-service):

* Every service runs Hibernate with `spring.jpa.hibernate.ddl-auto=validate`. Hibernate never
  creates or alters tables; it only validates that the entity mappings match the existing
  schema. If a mapping diverges from the schema, the service refuses to start.
* Each service maps *only* the tables it owns (see the ownership table in Section 2). The schema
  is *not* regenerated by any service.
* `db/01_schema.sql` is the *frozen, authoritative schema*. It is a read-only reference; it is
  never edited by the migrated code.
* Data a service does not own is *fetched over HTTP* from the owning service — never via a SQL
  join or sub-select that crosses a domain boundary. For example, `orders-service` calls
  `marketplace-service` for a unit price and `users-service` for a member tier, even though those
  rows are physically reachable in the same database.

This is why the cross-service HTTP contracts in Section 4 exist: they are the only legal channel
for cross-domain reads.

== 4. Cross-service HTTP contracts

Cross-domain communication is *HTTP-only*. No Java class in one service may import, instantiate,
or call a class from another service; the aggregator POM declares *no* inter-module compile-scope
dependency. The only legal channel is a set of thin `@Component` client gateways. Each contract
has a *producer* DTO in the owning service and a *separate consumer* DTO in the caller — the
duplication is intentional and avoids a shared module, thereby enforcing the boundary rule.

There are three contracts:

*Contract 1 — Pricing (`orders-service` -> `marketplace-service`)*

[source,http]
----
GET {marketplace.base-url}/api/products/{productId}/price?vendorId={vendorId}&qty={qty}

200 OK   -> a bare BigDecimal JSON number (no wrapper), e.g. 9.1692
404      -> InventoryNotFoundException
5xx      -> ServiceUnavailableException
----

*Contract 2 — Tier (`orders-service` -> `users-service`)*

[source,http]
----
GET {users.base-url}/api/members/{memberId}/tier

200 OK   -> {"tier":"GOLD"}   (BRONZE | SILVER | GOLD | PLATINUM)
404      -> MemberNotFoundException
5xx      -> ServiceUnavailableException
----

The producer DTO `MemberTierResponse` lives in `users-service`; the consumer DTO `MemberTierDto`
lives in `orders-service`.

*Contract 3 — Spend (`users-service` -> `orders-service`)*

[source,http]
----
GET {orders.base-url}/internal/members/{memberId}/spend?days={days}
X-Internal-Api-Key: {INTERNAL_API_KEY}

200 OK   -> {"totalSpend":2750.00}
401      -> {"error":"Unauthorized"}   (missing or invalid X-Internal-Api-Key)
404      -> BigDecimal.ZERO
5xx      -> ServiceUnavailableException
----

This endpoint is exposed under `/internal` because it is consumed only by another service, not by
the public frontend. Unlike the public `/api/**` contracts (1 and 2), which are reachable without
authentication, the `/internal/**` channel is guarded by a shared-secret filter: every request
must carry an `X-Internal-Api-Key` header whose value equals the server's `internal.api-key`
(sourced from the `INTERNAL_API_KEY` environment variable). The filter *fails closed* — if
`INTERNAL_API_KEY` is unset or blank, or the header is missing or does not match, the request is
rejected with `401 {"error":"Unauthorized"}`. The `users-service` consumer sends this header
automatically on the tier-recalculation spend read.

The producer DTO `MemberSpendResponse` lives in `orders-service`; the consumer DTO
`MemberSpendDto` lives in `users-service`.

== 5. Stored-procedure to Java parity

The six PL/pgSQL stored procedures in `db/02_stored_procedures.sql` have been reimplemented in
*pure Java* inside the owning `@Service` beans. *Zero* `EntityManager.createNativeQuery` call
sites remain after the migration; the stored procedures are never invoked by the application
(they are retained in the SQL file only as read-only references and as legacy parity fixtures for
the integration tests).

.Stored-procedure to Java parity matrix
[cols="2,2,4",options="header"]
|===
| Stored procedure | Java target (service.method) | Core logic reproduced

| `calculate_price`
| `marketplace` `PricingService`
| `unit = ROUND(base × (1 + markup/100) × (1 − volDisc), 4)`; volume discount tiers
  qty >= 100 -> .15, >= 50 -> .10, >= 20 -> .05, >= 10 -> .02, else 0; missing vendor inventory
  raises `InventoryNotFoundException` (HTTP 404, replacing the SQL `P0001`).

| `select_best_vendor`
| `marketplace` `VendorSelectionService`
| Scores eligible vendors and picks the best. *Source-A override (a)* — see below.

| `apply_customer_discount`
| `orders` `DiscountService`
| Tier rates PLATINUM .12 / GOLD .08 / SILVER .05 / BRONZE .02; `amount = ROUND(base × pct, 2)`;
  inserts exactly *one* `discount_audit` row per call — including non-committing previews (a
  quirk that is preserved, not "fixed").

| `calculate_shipping`
| `orders` `ShippingService`
| Resolves the zone by the first three ZIP digits (`start <= prefix <= end ORDER BY id LIMIT 1`);
  cost = `GREATEST(5.99, rate × weight)`; expedited orders apply a `× 2.5` surcharge *after* the
  5.99 floor; a null zone falls back to `$1.50/lb`; result rounded to 2 decimals.

| `process_order`
| `orders` `OrderService`
| Validates the member, validates a non-empty cart, inserts a `CONFIRMED` order, selects a vendor
  and prices each draft item into `order_items`, accumulates subtotal and weight, applies the
  discount, computes shipping, and sets `total = subtotal − discount + shipping`. Increments
  `member.total_spend`. *Source-A override (b)* — see below.

| `recalculate_customer_tiers`
| `users` `TierRecalculationService`
| For each member, sums the trailing 90-day `CONFIRMED` order totals; thresholds 5000 / 2000 / 500
  map to PLATINUM / GOLD / SILVER, otherwise BRONZE. Runs as `@Scheduled(cron="0 0 2 * * *")`
  plus an `@EventListener(ApplicationReadyEvent)` startup trigger, retaining single-node semantics.
|===

=== Source-A overrides

Where the prompt (Source A) and the SQL (Source B) disagree, *Source A wins*. Two overrides apply:

[source,text]
----
(a) Vendor selection uses a MAXIMIZATION score (pick the HIGHEST):
        score = (1/normPrice)*0.60 + (rating/5)*0.30 + (1/avgShippingDays)*0.10
    NOT the SQL minimization
        (price/minPrice)*0.6 + (1 - rating/5)*0.3 + (shipDays/10)*0.1  ORDER BY ... ASC

(b) member.total_spend increments by the order SUBTOTAL (not the SQL's total).
----

=== Canonical pricing example

The canonical pricing conformance check exercises `calculate_price` with base price `8.49` and
markup `8%`:

[source,text]
----
qty = 1   -> volDisc 0     -> 8.49 × 1.08 × (1 − 0)    = 9.1692
qty = 100 -> volDisc 0.15  -> 8.49 × 1.08 × (1 − 0.15) = 7.7938
----

These two values — *9.1692* at quantity 1 and *7.7938* at quantity 100 — are asserted by the
`marketplace-service` integration tests.

== 6. Prerequisites & Build

Tooling required to build the project:

* *JDK 17* (Eclipse Temurin) — the Spring Boot 3.x baseline. Do not use Java 21 / Spring Boot 4.x.
* *Apache Maven 3.9* (3.6.0+ is the documented minimum).
* *PostgreSQL 16* — the shared database (see Section 7).
* *Docker* — required for the Testcontainers-backed integration tests that `mvn clean install`
  (and `mvn verify`) run: each `*IT` test starts an ephemeral `postgres:16` container. Without a
  running Docker daemon the integration-test phase fails (see Section 8).
* *Node.js 20* — required only for the Playwright end-to-end suite under `e2e/`.

Build all three modules from the `kitchensink/` directory:

[source,bash]
----
cd kitchensink
mvn clean install
----

The aggregator parent (`packaging=pom`) builds the three child modules
(`marketplace-service`, `users-service`, `orders-service`) in order, producing one executable
Spring Boot JAR per service under each module's `target/` directory. The build declares no
inter-module compile-scope dependency, preserving the cross-domain boundary.

== 7. Running the services

=== Provision the database

Create the `kitchensink` database and load the three SQL scripts in order (schema, then stored
procedures, then seed data):

[source,bash]
----
# createdb / psql connect to the local PostgreSQL as the 'postgres' superuser on
# localhost:5432. Export PGPASSWORD (or rely on your local auth) so the client does not prompt.
export PGPASSWORD=postgres
createdb -h localhost -U postgres kitchensink
psql -h localhost -U postgres -d kitchensink -f db/01_schema.sql
psql -h localhost -U postgres -d kitchensink -f db/02_stored_procedures.sql
psql -h localhost -U postgres -d kitchensink -f db/03_seed_data.sql
----

NOTE: The stored procedures are loaded only so that legacy parity fixtures exist; the migrated
Java application never calls them.

NOTE: The seed data ships *no orders*, so each member's rolling 90-day `CONFIRMED`-order spend is
zero. When `users-service` starts, its `@EventListener(ApplicationReadyEvent)` tier recalculation
(Section 5) therefore assigns *every* freshly-seeded member the *BRONZE* tier — this recalculated
value, not the static `tier` column in `03_seed_data.sql`, is what `GET /api/members/{id}/tier`
returns and what the storefront badge and checkout discount reflect. Higher tiers (SILVER / GOLD /
PLATINUM) appear only once a member accumulates qualifying 90-day spend; the `TierRecalculationIT`
integration test seeds such orders to exercise those thresholds.

=== Start each service

Each service reads its datasource credentials from the environment — no secrets are committed to
`application.properties` (CWE-798), so `spring.datasource.username` / `spring.datasource.password`
resolve from `SPRING_DATASOURCE_USERNAME` / `SPRING_DATASOURCE_PASSWORD`. These two variables must
be exported before launching, or the service fails to start (Spring passes the unresolved
placeholder as the DB username and PostgreSQL rejects the login). Set them to the credentials of
your local `kitchensink` database (for example `postgres` / `postgres`), then run each module's
executable JAR (each starts an embedded Tomcat on its own port and context path):

[source,bash]
----
# Datasource credentials are read from the environment (no committed secrets).
export SPRING_DATASOURCE_USERNAME=postgres
export SPRING_DATASOURCE_PASSWORD=postgres

java -jar marketplace-service/target/*.jar   # http://localhost:8081/marketplace
java -jar orders-service/target/*.jar         # http://localhost:8082/orders
java -jar users-service/target/*.jar          # http://localhost:8083/users
----

=== Per-service configuration

Each service externalizes its configuration in `src/main/resources/application.properties`. Every
file sets `spring.datasource.url`, `spring.jpa.hibernate.ddl-auto=validate`, `server.port`, and
`server.servlet.context-path`; the services that make cross-service calls also set the relevant
`*.base-url` client properties.

`marketplace-service`:

[source,properties]
----
server.port=8081
server.servlet.context-path=/marketplace
spring.datasource.url=jdbc:postgresql://localhost:5432/kitchensink
spring.jpa.hibernate.ddl-auto=validate
----

`orders-service` (calls marketplace for pricing and users for tier):

[source,properties]
----
server.port=8082
server.servlet.context-path=/orders
spring.datasource.url=jdbc:postgresql://localhost:5432/kitchensink
spring.jpa.hibernate.ddl-auto=validate
marketplace.base-url=http://localhost:8081/marketplace
users.base-url=http://localhost:8083/users
----

`users-service` (calls orders for 90-day spend):

[source,properties]
----
server.port=8083
server.servlet.context-path=/users
spring.datasource.url=jdbc:postgresql://localhost:5432/kitchensink
spring.jpa.hibernate.ddl-auto=validate
orders.base-url=http://localhost:8082/orders
----

== 8. Testing

=== Integration tests

The integration tests are `@SpringBootTest` tests backed by Testcontainers. Each test spins up an
ephemeral `PostgreSQLContainer` seeded from the three `db/*.sql` scripts (`01_schema.sql`,
`02_stored_procedures.sql`, `03_seed_data.sql`) — the procedures are loaded only as legacy parity
fixtures and are never invoked by the code under test. The container's JDBC URL is wired into
Spring via `@DynamicPropertySource`. Cross-service calls are isolated by replacing each client
gateway interface (`MarketplaceClient`, `UsersClient`, `OrdersClient`) with a Mockito stub via
Spring Boot's `@MockitoBean`, so a service is tested without its downstream peers running.

Run the Java integration-test suite from `kitchensink/`. The `*IT` tests are bound to the Maven
Failsafe plugin (`integration-test` / `verify` goals), so use `mvn verify` — a plain `mvn test`
runs Surefire only and executes *none* of them. A running Docker daemon is required (Testcontainers
starts an ephemeral `postgres:16` per test):

[source,bash]
----
mvn verify
----

Parity assertions enforced by the integration tests:

* *Pricing* — `9.1692` at quantity 1 and `7.7938` at quantity 100 (base `8.49`, markup `8%`).
* *Order processing* — `member.total_spend` increments by the order *subtotal* (Source-A override b).
* *Discount audit* — exactly *one* `discount_audit` row is written per discount calculation,
  *including* non-committing preview calls.

The seven integration tests live under each owning service: `PricingServiceIT` and
`VendorSelectionServiceIT` (marketplace); `MemberRegistrationIT` and `TierRecalculationIT` (users);
`OrderServiceIT`, `DiscountServiceIT`, and `ShippingServiceIT` (orders).

=== End-to-end tests

A Playwright end-to-end suite lives under `e2e/`. It drives the PHP storefront against the three
running services, so *start all three services first* (Section 7). `npm test` runs a `pretest`
health gate (`wait-on` the three `/actuator/health` URLs) and then Playwright launches the PHP
storefront itself via its `webServer` block — `php -S 127.0.0.1:8080 -d expose_php=0 -t ..
../router.php` — so you do *not* start PHP manually for the e2e run. Chromium must be installed
once before the first run:

[source,bash]
----
cd e2e
npm ci                     # or: npm install
npm run install:browsers   # playwright install --with-deps chromium
npm test                   # pretest health-gate -> playwright test
----

To browse the storefront manually (outside the e2e run), start the three services, then serve the
PHP frontend from `kitchensink/` with the same router. `router.php` returns a real HTTP 404 for
unknown paths (instead of the built-in server's index fallback) and `-d expose_php=0` suppresses
the `X-Powered-By` header:

[source,bash]
----
# from kitchensink/ , with the three services already running
php -S 127.0.0.1:8080 -d expose_php=0 -t . router.php
# then open http://localhost:8080/frontend/index.php
----

== 9. Docker & CI

=== Docker

Each service ships with a multi-stage `Dockerfile` that builds with Maven and runs on a slim JRE:

* *Build stage* — `maven:3.9-eclipse-temurin-17` compiles the module and produces the executable JAR.
* *Runtime stage* — `eclipse-temurin:17-jre-alpine` runs the JAR.

Each `Dockerfile`'s `COPY` paths are relative to the aggregator directory (the child POMs inherit
from the aggregator parent), so the build *context* must be `kitchensink/` and the Dockerfile is
selected with `-f`. Build a service image (run from `kitchensink/`):

[source,bash]
----
docker build -f marketplace-service/Dockerfile -t net32/marketplace-service .
docker build -f orders-service/Dockerfile     -t net32/orders-service .
docker build -f users-service/Dockerfile       -t net32/users-service .
----

=== Continuous integration

A GitHub Actions workflow at `.github/workflows/ci.yml` builds and tests the whole project on
every push. The job provisions a `postgres:16` service container for the database, builds all
three modules with Maven, runs the `@SpringBootTest` + Testcontainers integration tests, and then
executes the Playwright end-to-end suite under `e2e/`.

NOTE: The pre-existing `.github/workflows/reduce_readme.yml` documentation workflow is unrelated
to the build and is left unchanged.
