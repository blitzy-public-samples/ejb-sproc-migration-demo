= kitchensink: Spring Boot 3.x microservices (marketplace, orders, users)
:author: Pete Muir
:level: Intermediate
:technologies: Spring Boot, Spring Data JPA, Spring MVC, Bean Validation, Testcontainers

[abstract]
The `kitchensink` quickstart, originally a single Jakarta EE WAR, is now decomposed into three independently deployable Spring Boot 3.x (Java 17) microservices that communicate over HTTP. Business logic that formerly lived in PostgreSQL stored procedures now lives in plain Java service classes.

== What is it?

`kitchensink` started life as a single Jakarta EE web application whose core business rules were concentrated in PL/pgSQL stored procedures and invoked through native queries. It has been refactored — strictly along business-domain lines — into three self-contained Spring Boot 3.x services. Each service owns its domain entities, repositories, services, and REST surface; the extracted procedure logic now runs in Java; and every cross-domain interaction is a plain HTTP call.

[cols="1,1,2,4", options="header"]
|===
| Service | Port | Context-path | Responsibilities

| `marketplace-service`
| 8081
| `/marketplace`
| Product catalog, pricing, and vendor selection.

| `orders-service`
| 8082
| `/orders`
| Shopping cart, order orchestration, discounting, and shipping.

| `users-service`
| 8083
| `/users`
| Member registration and loyalty-tier management.
|===

[NOTE]
====
The three services share one physical PostgreSQL database, but each Hibernate persistence context maps only the tables it owns and runs with `spring.jpa.hibernate.ddl-auto=validate` against the externally managed schema. Tables owned by another service are simply left unmapped, so per-service validation succeeds. There are no shared Java classes and no inter-service compile-time dependency: all cross-service interaction happens over HTTP through thin client components.
====

== System Requirements

* *Java 17 JDK* — the Spring Boot 3.x baseline runtime.
* *Maven 3.9+* — to build the multi-module reactor and repackage each service as an executable JAR.
* *Docker* — required by the Testcontainers-based integration tests and to build the per-service container images.
* *PostgreSQL* — a database named `kitchensink`, reachable at `localhost:5432`, holding the application schema and seed data (see the Database setup section below).

== Database setup

Create a PostgreSQL database named `kitchensink`, then load the authoritative schema followed by the seed data — for example with `psql` from inside the `kitchensink/` directory:

[source,bash]
----
psql -h localhost -U "$DB_USERNAME" -d kitchensink -f db/01_schema.sql
psql -h localhost -U "$DB_USERNAME" -d kitchensink -f db/03_seed_data.sql
----

[NOTE]
====
`db/02_stored_procedures.sql` is retained only as *reference documentation*. The migrated Spring Boot services re-implement that logic in Java and never invoke these stored procedures, so the file must *not* be loaded into the running database.
====

== Building the quickstart

From inside the `kitchensink/` directory, build all three modules and run their integration tests in a single reactor build:

[source,bash]
----
cd kitchensink
mvn -B clean verify
----

This compiles `marketplace-service`, `orders-service`, and `users-service`, repackages each as an executable Spring Boot fat JAR, and runs the integration tests. A running Docker daemon is required, because the integration tests start a disposable PostgreSQL container via Testcontainers.

== Running the services

Each service is a self-contained executable JAR. Supply the database credentials through the `DB_USERNAME` and `DB_PASSWORD` environment variables, then start each service (typically one per terminal):

[source,bash]
----
export DB_USERNAME=<your-db-user>
export DB_PASSWORD=<your-db-password>

# marketplace-service -> http://localhost:8081/marketplace
java -jar marketplace-service/target/marketplace-service-*.jar

# orders-service -> http://localhost:8082/orders
java -jar orders-service/target/orders-service-*.jar

# users-service -> http://localhost:8083/users
java -jar users-service/target/users-service-*.jar
----

Cross-service calls are configured through `services.*.base-url` properties, whose defaults already point at the local ports and context-paths shown above:

* *orders-service* calls both peers, so it reads `services.marketplace.base-url` (default `http://localhost:8081/marketplace`) and `services.users.base-url` (default `http://localhost:8083/users`).
* *users-service* calls orders-service during loyalty-tier recalculation, so it reads `services.orders.base-url` (default `http://localhost:8082/orders`).
* *marketplace-service* makes no outbound service calls.

When the services run on different hosts, override any base URL with the matching environment variable — for example `SERVICES_MARKETPLACE_BASE_URL`, `SERVICES_USERS_BASE_URL`, or `SERVICES_ORDERS_BASE_URL`.

== Verifying the services

Each service exposes a Spring Boot Actuator health endpoint under its context-path. Once a service is up, the health endpoint responds with HTTP 200 and a JSON body whose `status` field is `UP`:

[source,bash]
----
curl http://localhost:8081/marketplace/actuator/health
curl http://localhost:8082/orders/actuator/health
curl http://localhost:8083/users/actuator/health
# each responds with: {"status":"UP"}
----

The original REST endpoints are preserved, each under its service's base mapping. Example calls:

NOTE: The marketplace-service product-catalog endpoint (`GET /marketplace/api/products`) is served by the marketplace producer controller (`ProductResourceRESTService`), which is delivered in the final step of the migration. Until that controller is in place, the marketplace-service still boots and its `/marketplace/actuator/health` endpoint responds `UP`, but `GET /marketplace/api/products` is not yet available. The orders-service and users-service endpoints shown below are available now.

[source,bash]
----
# marketplace-service — product catalog (available after the final marketplace producer controller is in place)
curl http://localhost:8081/marketplace/api/products

# orders-service — look up an order by id
curl http://localhost:8082/orders/api/orders/{id}

# users-service — list members, or register a new member
curl http://localhost:8083/users/api/members
curl -X POST http://localhost:8083/users/api/members \
     -H 'Content-Type: application/json' \
     -d '{"name":"Jane Doe","email":"jane.doe@example.com","phoneNumber":"2125551234"}'
----

== Running the integration tests

`mvn -B clean verify` (run from inside `kitchensink/`) executes the migrated integration tests. They are `@SpringBootTest` tests backed by a Testcontainers-managed PostgreSQL instance — no application server is required. The Testcontainers database is initialized from `db/01_schema.sql` and `db/03_seed_data.sql` only; downstream HTTP dependencies are stubbed in-process, so the tests run without the peer services running. The tests are distributed to their owning services:

* *marketplace-service* — `PricingServiceIT`
* *orders-service* — `DiscountServiceIT`, `OrderServiceIT`
* *users-service* — `MemberRegistrationIT`, `RemoteMemberRegistrationIT`, `TierRecalculationIT`

The repository-root workflow `.github/workflows/ci.yml` runs the same `mvn -B clean verify` on a Docker-enabled runner for every push and pull request.

== Container images

Each service ships a multi-stage `Dockerfile` (a `maven:3.9-eclipse-temurin-17` build stage that produces the fat JAR, and an `eclipse-temurin:17-jre-alpine` runtime stage). Build and run an image as follows (shown for `marketplace-service`; repeat for the others, mapping ports `8082` and `8083`):

[source,bash]
----
docker build -t marketplace-service marketplace-service/
docker run --rm -p 8081:8081 \
  -e DB_USERNAME=<your-db-user> \
  -e DB_PASSWORD=<your-db-password> \
  marketplace-service
----

The container image build skips the integration tests (they require a Docker daemon for Testcontainers, which is unavailable inside the build); test execution lives in CI instead.

== A note on the former web UI

The legacy JSF member-registration UI was removed together with the original WAR packaging. The equivalent functionality is now available through the users-service REST endpoint (`POST http://localhost:8083/users/api/members`). The PHP storefront under `frontend/` is retained and integrates with the three services purely over HTTP, using the per-service base URLs configured in `frontend/includes/config.php`.
